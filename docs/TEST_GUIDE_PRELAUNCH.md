# ローンチ前テストガイド（テスト環境 → 本番環境を見据えて）

このドキュメントは、**テスト環境**で一通りチェックしたあと、**本番環境（Vercel・Supabase を別プロジェクトで用意する想定）** に進むための手順と、システム開発で把握しておくとよいポイントをまとめたものです。

---

## 前提の整理

| 環境 | 用途 | Vercel | Supabase |
|------|------|--------|----------|
| **テスト環境** | 今ここで検証している環境 | テスト用プロジェクト | テスト用プロジェクト |
| **本番環境** | 実際の顧客向け | 本番用プロジェクト（別） | 本番用プロジェクト（別） |

- テスト環境で「OK」と判断できたら、本番用に **Vercel と Supabase を新規プロジェクトで用意** し、同じコードベースをデプロイ・接続する流れになります。
- 環境変数は **環境ごとに別々に設定** します（テスト用と本番用で値が違って当然です）。

---

## Step 0: テスト前の掃除（推奨）

テスト前に「使っていないもの」や「本番に不要なもの」を整理しておくと、**本番デプロイ時の混乱を防げます**。必須ではありませんが、**やった方がよい**です。

### やった方がよいもの

| 対象 | 説明 | やり方 |
|------|------|--------|
| **一時メモ・一時ファイル** | 開発中のメモでリポジトリに残さないもの | 不要なファイルを削除する |
| **テスト専用ルート** | `src/app/test/route.ts` と `src/app/api/test/route.ts` は Supabase 接続確認用。テスト中は便利だが、**本番では公開しない方がよい**（DB の存在や件数がわかる） | テスト環境では残してOK。本番デプロイ前に削除するか、環境変数で本番では 404 を返すようにする |

### やらなくてよいもの

| 対象 | 理由 |
|------|------|
| **scripts/**（例: `list-busy-season-holiday-blocks.ts`） | 運用・開発用のスタンドアロンスクリプト。ビルドには含まれず、本番アプリの動作にも影響しない。残しておいてよい |
| **Supabase トリガー・関数の整理**（`supabase/TRIGGERS_AND_FUNCTIONS.md` および `supabase/manual/definitions/`） | トリガー・RPC の一覧と実機定義。運用・本番移行時に参照するため **残しておく** |
| **削除済みの `supabaseClient.ts`** | 既に削除されており、コードはすべて `@/utils/supabase/server` 等を参照している。対応不要 |

### データの掃除（Supabase テスト環境）

- **テスト環境の Supabase** に、過去の検証用予約やダミーデータが残っている場合、**テスト前に消しておくと「まっさらな状態」でフローを確認**できます。
- やり方: Supabase ダッシュボード → Table Editor で該当テーブル（例: `reservations`）の行を削除するか、SQL Editor で `DELETE FROM reservations WHERE ...` や、**テスト用プロジェクトだけ**であれば `TRUNCATE` を実行する。
- **注意**: 本番用 Supabase では絶対に `TRUNCATE` しないこと。テスト環境と本番は別プロジェクトなので、テスト環境のデータ削除は本番に影響しません。

## Step 1: ローカル環境の動作確認

### 1-1. 依存関係のインストール

**やること**

プロジェクトのルート（`kisocycle-test` フォルダ）で、ターミナルを開いて次を実行します。

```bash
npm install
```

**何をしているか**

- `package.json` に書かれたライブラリ（Next.js、Supabase、Resend など）を `node_modules` にダウンロードしています。
- 初回や `package.json` を変えたあとは必ず実行します。

**把握しておくこと**

- **依存関係（dependencies）**: 本番でも必要なパッケージ。`package.json` の `dependencies` に書く。
- **開発用依存関係（devDependencies）**: ビルド・型チェック用など、開発時だけ使うもの。`devDependencies` に書く。
- `npm install` は両方を入れます。本番デプロイ時（Vercel など）は通常 `npm install --production` 相当になり、`devDependencies` は入れないことが多いです。

---

### 1-2. 環境変数の準備

**やること**

プロジェクトのルートに `.env.local` というファイルを作成し、次の変数を書きます。  
（`.env.local` は Git に含めないので、自分だけの設定として使います。）

**テンプレート（コピー用）**

```bash
# ===== Supabase（テスト用プロジェクトのURLとキー） =====
NEXT_PUBLIC_SUPABASE_URL=https://xxxxxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
SUPABASE_SERVICE_ROLE_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

# ===== 管理画面の Basic 認証（テスト用のユーザー名・パスワード） =====
ADMIN_BASIC_USER=admin
ADMIN_BASIC_PASSWORD=ここに強いパスワードを書く

# ===== メール送信（Resend）※テスト時はなくても予約はできるがメールは送れない =====
RESEND_API_KEY=re_xxxxxxxxxxxx
RESEND_FROM_EMAIL=noreply@あなたのドメイン.com

# ===== 任意（メール内リンクのベースURL。Vercel デプロイ後は設定推奨） =====
# NEXT_PUBLIC_BASE_URL=https://あなたのテスト用.vercel.app
```

**値の取り方（テスト環境用）**

1. **Supabase**  
   - [Supabase ダッシュボード](https://supabase.com/dashboard) → テスト用プロジェクト → **Settings → API**  
   - **Project URL** → `NEXT_PUBLIC_SUPABASE_URL`  
   - **anon public** → `NEXT_PUBLIC_SUPABASE_ANON_KEY`  
   - **service_role** → `SUPABASE_SERVICE_ROLE_KEY`（絶対に外部に漏らさない）

2. **ADMIN_BASIC_USER / ADMIN_BASIC_PASSWORD**  
   - 管理画面（`/admin`）にログインするときのユーザー名とパスワード。テスト用に好きな文字列でOK（本番では長く複雑に）。

3. **Resend**  
   - [Resend](https://resend.com) で API キーを発行。  
   - `RESEND_FROM_EMAIL` は Resend で認証した送信元アドレス。  
   - 未設定でも予約は完了しますが、確認メールは送信されません。

**把握しておくこと**

- **NEXT_PUBLIC_ がつく変数**: ブラウザに露出するので、秘密キーを入れない。
- **つかない変数**: サーバーだけで使う。API キーや service_role はここに入れる。
- **本番環境**では、Vercel の「Environment Variables」と Supabase の本番プロジェクトで、上と同じ名前で**別の値**を設定します。

---

### 1-3. ビルドが通るか確認

**やること**

```bash
npm run build
```

**期待すること**

- 途中でエラーが出ず、最後に「Compiled successfully」や `Route (app)` 一覧が出る。
- `Type error` や `Module not found` が出たら、コードか環境変数の不足を疑います。

**把握しておくこと**

- **ビルド**: ソースコードを本番用にまとめて、`/.next` などに出力する処理です。
- ここが通らないと Vercel でもデプロイできません。**「動かす前の必須チェック」** と思ってよいです。
- 環境変数は、ビルド時に「存在するか」は読まれても、**値は実行時（`npm run start` や Vercel 上）で読む**ので、ビルド時にはダミーでそろえておけば通る場合があります（Supabase の URL だけ入れて試すなど）。
- **型エラー**: TypeScript の型チェックで、`string | null` を `string` だけ受け付ける関数に渡すとビルドが失敗します。`end_date ?? start_date` のように「null のときの fallback」を渡すと解消します（`pannier-availability` で実施した修正と同じ考え方）。
- **npm audit**: `npm install` 後に「vulnerability」と出ることがあります。本番前に `npm audit` で内容を確認し、必要なら `npm audit fix` で対応します。テスト環境ではまずビルド通過を優先してOKです。

## Step 2: ユーザー予約フローの手動テスト

**目的**: お客様がトップページから「日付・車種・オプションを選ぶ → 内容確認 → 予約する」まで一通り動くか確認します。システムを作るうえで「**ユーザーが触る画面を自分で触ってみる**」ことはとても重要です。

### 2-1. 開発サーバーを起動する

**やること**

ターミナルでプロジェクトのルートにいる状態で、次を実行します。

```bash
npm run dev
```

**期待すること**

- `▲ Next.js 15.x.x` と表示され、`Local: http://localhost:3000` のような URL が出る。
- ブラウザで **http://localhost:3000** を開くと、トップページ（レンタサイクル予約フォーム）が表示される。

**把握しておくこと**

- **開発サーバー（dev）**: コードを変更すると自動で反映されるモード。本番用のビルド（`npm run build`）とは別で、**開発中は `npm run dev` で動かす**のが基本です。
- **本番モードで動かしたい場合**: `npm run build` のあと `npm run start` で、本番と同じ動きをローカルで確認できます。テストでは `npm run dev` で十分です。

---

### 2-2. トップページで「予約フォーム」を操作する

**やること**

1. **http://localhost:3000** を開く。
2. **利用日**（貸出日・返却日）を選ぶ。  
   - 今日より前の日付は選べないこと、**水曜（定休日）は貸出日として選べない**こと（返却日は水曜可）を確認する。
3. **プラン**（6時間 / 1日 / 2日以上）を選ぶ。  
   - 繁忙期の三連休では「6時間プラン」が選べない制約がある場合は、その旨表示されることを確認する。
4. **車種と台数**を選ぶ。  
   - 複数車種を選べる。在庫がある組み合わせだけ「予約可能」になることを確認する。
5. **オプション**（スマホホルダー、パニアバッグなど）を必要に応じて選ぶ。  
   - パニアバッグは「全予約合計で最大数」の制限があるため、上限を超えるとエラーや注意が出ることを確認する。
6. **料金を確認**し、「予約内容を確認」ボタンを押す。
7. **最終確認画面**で、名前・メールアドレス・電話番号を入力する。  
   - テスト用に自分宛てのメールアドレスを入れてよい。
8. **「この内容で予約する」** をクリックする。

**期待すること**

- エラーが出ずに「ご予約が完了しました」のような**完了画面**に遷移する。
- **予約番号**（8文字程度の英数字）が表示される。
- 完了画面に「ご予約内容はご登録のメールアドレスにも送信されています」「キャンセル申請リンクからお手続きを」などの案内が出る。
- **Resend を設定している場合**: 入力したメールアドレスに**確認メール**が届く。メール本文に**キャンセル申請用リンク**（`/cancel/xxxxxxxx` のような URL）が含まれていることを確認する。

**把握しておくこと**

- **フロントエンドとAPI**: 画面上のボタンを押すと、裏で `POST /api/reserve` が呼ばれ、Supabase に予約が保存されます。**「画面の操作」と「API・DB」がこうやってつながっている**と意識しておくと、今後のシステム作りで役立ちます。
- **在庫チェック**: 車種・日付に応じて在庫が API（`/api/availability` や `/api/pannier-availability`）でチェックされています。在庫が足りないと「予約不可」やボタンが押せない動きになります。
- **メールが届かない場合**: Resend の API キーや送信元アドレスが未設定・不正だと、予約は完了してもメールは送られません。テスト時は `.env.local` の `RESEND_*` を確認し、Resend ダッシュボードで送信ログも見ると原因を追いやすいです。

- [ ] 日付・プラン・車種・オプションを選び、最終確認まで進めた
- [ ] 「この内容で予約する」で完了画面が出て、予約番号が表示された
- [ ] （Resend 設定済みなら）確認メールが届き、キャンセル用リンクが含まれていた

## Step 3: 管理画面の確認

**目的**: 管理者が予約一覧・在庫・キャンセル承認などを**管理画面（/admin）**で操作できることを確認します。**「誰が・どの画面で・何をするか」**を分けて考えると、権限設計や今後の機能追加がしやすくなります。

### 3-1. 管理画面にログインする

**やること**

1. ブラウザで **http://localhost:3000/admin/reservations** を開く（またはトップから管理画面へのリンクがあればそこから）。
2. **Basic 認証**のポップアップが出るので、`.env.local` に設定した **ADMIN_BASIC_USER** と **ADMIN_BASIC_PASSWORD** を入力する。
3. 認証に成功すると「予約管理」の画面が表示される。

**期待すること**

- ユーザー名・パスワードが正しければ予約一覧（または「該当する予約がありません」）が表示される。
- 誤ったパスワードだと「認証が必要です」となり、一覧が見られない。

**把握しておくこと**

- **Basic 認証**: このプロジェクトでは `src/middleware.ts` が `/admin` および `/api/admin` へのアクセスをチェックし、`ADMIN_BASIC_USER` / `ADMIN_BASIC_PASSWORD` と一致する場合だけ通過させています。**本番では必ず強いパスワードを設定**し、Vercel の環境変数にも同じ名前で入れます。
- **ミドルウェア**: Next.js の `middleware` は「リクエストがページや API に届く前」に実行されます。認証・リダイレクト・ヘッダー追加など、**全ルートで共通したい処理**をここに書く、と覚えておくとよいです。

---

### 3-2. 予約一覧・ステータス・キャンセル承認を確認する

**やること**

1. **予約一覧**で、Step 2 で作成した予約が一覧に表示されていることを確認する。  
   - 名前・メール・貸出日・返却日・プラン・金額・ステータス（予約済み など）が出ていれば OK。
2. **検索**（名前 / メール / 予約番号）で、該当する予約だけに絞れることを確認する。
3. **ステータス**が「予約済み（reserved）」になっていることを確認する。
4. （Step 4 でキャンセル申請をしたあと）一覧でその予約の「キャンセル申請あり」のような表示を確認し、**キャンセル承認**ボタンを押して承認する。  
   - 承認後、ステータスが「キャンセル済み」になり、在庫が復元される動きを確認する（在庫画面でも確認するとよい）。

**期待すること**

- 予約が一覧に表示され、検索・フィルタが動く。
- キャンセル申請 → 管理画面で承認 → キャンセル済みになり、在庫が戻る。

**把握しておくこと**

- **管理者だけが触る画面**: `/admin` は一般ユーザーには公開せず、Basic 認証で守っています。**「お客様用」と「管理者用」で URL と権限を分けている**という設計です。
- **キャンセルは「申請 → 承認」の2段階**: お客様がキャンセル申請すると `cancel_requested` が立ち、管理者が承認すると実際にキャンセル（ステータス変更＋在庫復元）されます。トラブル防止や在庫管理のためによく使うパターンです。

---

### 3-3. 在庫画面（/admin/stock）を確認する

**やること**

1. **http://localhost:3000/admin/stock** を開く（同じ Basic 認証で入れる）。
2. 車種ごとの**在庫数**（マスタ在庫・予約済み・残りなど）が表で見えることを確認する。
3. 必要なら「電話・店頭予約」を反映する操作がある場合は、その動作も確認する。

**期待すること**

- 予約した車種の「予約済み」数が増えていること、キャンセル承認後に減っていることが分かる。

**把握しておくこと**

- **在庫は DB（Supabase）のマスタと予約から計算**されています。本番では「オンライン予約」と「電話・店頭」の両方をどう扱うかを決めておくと、運用が楽になります。

---

- [ ] 予約一覧に Step 2 の予約が表示され、検索・フィルタが使えた
- [ ] （Step 4 実施後）キャンセル申請を承認し、ステータスが「キャンセル済み」になり在庫が戻った
- [ ] /admin/stock で在庫数が確認できた

## Step 4: キャンセルフローの確認

**目的**: お客様が「確認メールのリンクからキャンセル申請 → 管理者が承認」まで一通り動くか確認します。**キャンセルは在庫に戻る**ので、この流れが正しく動くことが重要です。

### 4-1. キャンセル申請ページを開く

**やること**

1. Step 2 で予約完了後に届いた**確認メール**を開く（Resend 未設定の場合は、Step 2 の予約で使った**予約番号**を控えておく）。
2. メール本文中の **「キャンセル申請はこちら」** のようなリンクをクリックする。  
   - リンクは `https://.../cancel/xxxxxxxx` の形。ローカルでは `http://localhost:3000/cancel/xxxxxxxx` になる（メールのベースURLが `NEXT_PUBLIC_BASE_URL` で、未設定だと localhost になる）。
3. **ローカルでテストする場合**: メールが localhost を指していないときは、ブラウザで直接 **http://localhost:3000/cancel/（予約ID）** を開く。予約IDは管理画面の予約一覧や、Supabase の `reservations` テーブルの `id` で確認できる（先頭8文字で十分な場合が多い）。

**期待すること**

- キャンセル申請ページが開き、**予約内容**（予約番号・名前・プラン・貸出日・返却日・金額など）が表示される。
- 既にキャンセル済みの予約なら「この予約は既にキャンセルされています」と表示される。

**把握しておくこと**

- **メール内リンクのベースURL**: 本番や Vercel にデプロイしたあとは、`NEXT_PUBLIC_BASE_URL` に **https://あなたのドメイン.vercel.app** のように設定すると、メールのキャンセルリンクが本番URLを指します。ローカルでは未設定で localhost になるので、手動で `http://localhost:3000/cancel/（ID）` を開いてテストすれば十分です。

---

### 4-2. キャンセル申請を送信する

**やること**

1. キャンセル理由（任意）を入力する。
2. **「キャンセル申請する」** ボタンを押す。
3. 「キャンセル申請を承りました。お店で承認後、キャンセルが確定します」のような**申請完了画面**になることを確認する。
4. 同じ URL を再度開くと「キャンセル申請済み」と表示され、二重申請できないことを確認する。

**期待すること**

- 申請が送信され、申請済みの案内が出る。
- 管理画面（Step 3）でその予約に「キャンセル申請あり」が出て、**承認**できること。承認後、予約ステータスが「キャンセル済み」になり、在庫が復元される（在庫画面で確認）。

**把握しておくこと**

- **申請と確定を分ける**: お客様の「キャンセル申請」は即キャンセルではなく、**管理者の承認**で確定する設計です。在庫の取り違えや誤操作を防ぎやすく、今後のシステムでも「申請 → 承認」のパターンはよく使います。

---

### Step 4 のチェックリスト

- [ ] 確認メールのリンク、または手動で /cancel/（予約ID） を開き、予約内容が表示された
- [ ] キャンセル申請を送信し、「申請済み」の案内が出た
- [ ] 管理画面でキャンセル申請を承認し、ステータスが「キャンセル済み」になり在庫が戻った

ここまで OK なら、**Step 5（Vercel デプロイ後の確認・本番前チェックリスト）** に進みます。

## Step 5: Vercel にデプロイしてテストする（推奨）

**目的**: ローカルだけでなく、**Vercel にデプロイした URL** でも同じように動くか確認します。本番は「Vercel ＋ Supabase を別プロジェクトで用意」するため、**テスト環境でも一度 Vercel に上げておくと、本番移行時の手順に慣れます**。

### 5-1. Vercel にデプロイする

**やること**

1. [Vercel](https://vercel.com) にログインし、**このリポジトリをインポート**（GitHub 連携など）してプロジェクトを作る。
2. プロジェクトの **Settings → Environment Variables** で、`.env.local` と同じ**変数名と値**を設定する。  
   - テスト用 Supabase の URL・anon・service_role、ADMIN_BASIC_USER / ADMIN_BASIC_PASSWORD、Resend（任意）、**NEXT_PUBLIC_BASE_URL** には `https://あなたのプロジェクト名.vercel.app` を入れる（メールのキャンセルリンク用）。
3. **Deploy** する。main ブランチへの push で本番デプロイ、PR でプレビューデプロイになることが多いです。

**期待すること**

- ビルドが成功し、`https://xxxx.vercel.app` でサイトが開く。
- トップページ・予約フロー・管理画面（Basic 認証）・キャンセルページが、ローカルと同様に動く。

**把握しておくこと**

- **環境変数はデプロイ先ごと**: Vercel の「Environment Variables」は**そのプロジェクト専用**です。本番用に別プロジェクトを作ったら、本番用の Supabase など**別の値**を同じ変数名で設定し直します。
- **NEXT_PUBLIC_BASE_URL**: Vercel の URL を入れておくと、確認メールの「キャンセル申請はこちら」が Vercel の URL を指すようになり、お客様が正しいページを開けます。

---

### 5-2. デプロイ先 URL で一通りやり直す

**やること**

- **https://あなたの.vercel.app** で、Step 2〜4 と同様に  
  「予約する → 確認メールのリンクでキャンセル申請 → 管理画面で承認」  
  を一度実行する。
- メールのキャンセルリンクが **Vercel の URL** になっていることを確認する。

**期待すること**

- 予約・メール・キャンセルリンク・管理画面の承認まで、すべてデプロイ先で問題なく動く。

- [ ] Vercel にデプロイし、環境変数（と NEXT_PUBLIC_BASE_URL）を設定した
- [ ] デプロイ先の URL で予約 → メールのリンクでキャンセル申請 → 管理画面で承認まで確認した

## 本番環境に進む前のチェックリスト

テスト環境で上記がすべて問題なければ、**本番用の Vercel と Supabase を新規プロジェクトで用意**し、同じコードベースをデプロイ・接続する流れになります。その前に、次を確認してください。

| 項目 | 内容 |
|------|------|
| **テスト専用ルート** | `src/app/test/route.ts` と `src/app/api/test/route.ts` を本番では公開しない。削除するか、環境変数で「本番のときは 404」にする。 |
| **環境変数** | 本番用 Vercel プロジェクトで、本番用 Supabase の URL・anon・service_role、本番用の ADMIN_BASIC_USER / ADMIN_BASIC_PASSWORD、Resend、**NEXT_PUBLIC_BASE_URL**（本番ドメイン）を設定する。 |
| **Supabase** | 本番用に**新しいプロジェクト**を作り、マイグレーションやスキーマを適用する。テスト環境の DB をそのまま本番にしない。 |
| **パスワード** | 本番の ADMIN パスワードは長く複雑にする。Vercel の「Deployment Protection → Password Protection」でサイト全体を守る場合は `docs/VERCEL_PASSWORD_PROTECTION.md` を参照。 |
| **メール** | Resend の送信元ドメイン・API キーが本番用として問題ないか確認する。 |

## まとめ：テストの流れと今後につながるポイント

| Step | やること | 把握しておくとよいこと |
|------|----------|------------------------|
| **0** | テスト前の掃除（不要ファイル・テスト用データの整理） | 本番に不要なものはテスト環境のうちに整理しておく |
| **1** | ローカルで `npm install` → `.env.local` → `npm run build` | ビルドが通らないとデプロイできない。環境変数は「実行時」に読まれる |
| **2** | トップから予約完了まで手動で操作 | 画面の操作と API・DB がどうつながっているかを意識する |
| **3** | 管理画面で予約一覧・在庫・キャンセル承認 | 「お客様用」と「管理者用」で URL と認証を分けている設計 |
| **4** | メールのリンクからキャンセル申請 → 管理画面で承認 | キャンセルは「申請 → 承認」の2段階。在庫復元の流れを確認 |
| **5** | Vercel にデプロイして同じテストを実行 | 環境変数はデプロイ先ごとに設定する。本番は別プロジェクトで別の値 |

このガイドの手順でテスト環境を一通り確認できたら、本番用の Vercel・Supabase を用意し、同じコードベースで本番デプロイに進めます。システムを作るのが初めての場合も、**「ビルド → ローカルで動作確認 → デプロイ先で確認 → 本番用は別プロジェクトで」** の流れを一度経験しておくと、次のサービスづくりにそのまま活かせます。
